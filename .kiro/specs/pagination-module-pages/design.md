# 设计文档：原创IP世界观介绍网页多页应用改造

## 概述

本设计文档描述了将原创IP世界观介绍网页从单页应用（SPA）改造为多页应用（MPA）的技术方案。改造的核心目标是将9个模块分离为独立的HTML页面，同时保持原有的设计风格、交互功能和用户体验。

### 改造范围

- **主页（index.html）**: 包含导航栏、图片滚动、项目介绍
- **模块页面（9个）**: 地图、物体、事件、名词、组织、国度、其他设定、神话、世界时间线
- **共享资源**: CSS样式表、JavaScript脚本、图片文件、音乐文件

### 关键挑战

1. **相对路径管理**: 不同目录深度的页面需要正确加载共享资源
2. **状态跨页面保持**: 夜间模式、音乐播放状态需要在页面间保持
3. **导航一致性**: 所有页面需要相同的导航栏和高亮功能
4. **资源共享**: CSS、JavaScript、图片需要被所有页面正确加载

### 路径处理策略

采用**相对路径**方案，所有资源引用使用相对于HTML文件的路径：

- **主页（index.html）**: 资源路径为 `./css/style.css`、`./js/script.js`
- **模块页面（pages/\*.html）**: 资源路径为 `../css/style.css`、`../js/script.js`

这样可以确保无论用户访问哪个页面，资源都能被正确加载。

## 组件和接口设计

### 1. 导航栏组件

**功能**:

- 显示所有9个模块的链接
- 显示返回主页的链接
- 高亮显示当前页面
- 在所有页面上保持一致

**实现方式**:

- 使用HTML `<nav>` 元素
- 通过JavaScript检测当前页面URL，动态添加 `active` 类
- 导航栏HTML在每个页面中重复（或通过JavaScript动态注入）

**导航链接结构**:

```html
<nav class="nav-container">
  <h3 class="nav-title">世界观导航</h3>
  <ul class="nav-list">
    <li><a href="/" class="nav-link">主页</a></li>
    <li><a href="/pages/map.html" class="nav-link">地图</a></li>
    <li><a href="/pages/objects.html" class="nav-link">物体</a></li>
    <!-- ... 其他模块 ... -->
  </ul>
</nav>
```

### 2. 状态管理模块

**功能**:

- 管理夜间模式状态
- 管理音乐播放状态
- 跨页面持久化状态

**实现方式**:

- 使用 `localStorage` 存储用户偏好设置
- 创建 `state-manager.js` 模块，提供统一的状态管理接口
- 页面加载时从 `localStorage` 恢复状态

**状态存储结构**:

```javascript
{
  darkMode: boolean,        // 夜间模式状态
  musicPlaying: boolean,    // 音乐播放状态
  lastPage: string         // 上次访问的页面
}
```

### 3. 音乐播放管理

**功能**:

- 在页面间继续播放音乐
- 保持音乐播放状态

**实现方式**:

- 使用全局 `Audio` 对象，存储在 `window` 对象上
- 页面加载时检查是否已有音乐对象，如果有则继续使用
- 通过 `sessionStorage` 或 `window` 对象传递音乐状态

**关键代码**:

```javascript
// 在 shared.js 中
window.backgroundMusic =
  window.backgroundMusic || new Audio("../voice/2月28日.MP3");
```

### 4. 图片滚动组件

**功能**:

- 仅在主页显示
- 自动每7秒切换一次
- 支持手动切换

**实现方式**:

- 在主页中包含图片滚动HTML结构
- 在模块页面中不包含此组件
- JavaScript检测是否存在滚动容器，存在则初始化

## 正确性属性

_属性是系统在所有有效执行中应该保持为真的特征或行为——本质上是关于系统应该做什么的正式陈述。属性充当人类可读规范和机器可验证正确性保证之间的桥梁。_

### Property 1: 所有模块页面都能正确加载资源

_对于任何模块页面，当页面加载时，所有CSS、JavaScript和图片资源都应该被正确加载，不产生404错误。_

**验证需求: 需求 8.1, 8.2, 8.3, 8.4**

### Property 2: 导航栏在所有页面上保持一致

_对于任何页面，导航栏都应该包含所有9个模块的链接和返回主页的链接，且链接的href属性应该指向正确的页面。_

**验证需求: 需求 2.1, 3.1**

### Property 3: 当前页面在导航栏中被正确高亮

_对于任何页面，导航栏中对应该页面的链接应该具有 `active` 类，其他链接不应该具有此类。_

**验证需求: 需求 9.1**

### Property 4: 夜间模式状态在页面间保持

_对于任何页面，如果用户启用了夜间模式，当用户导航到其他页面时，夜间模式状态应该保持启用，且页面应该应用深色主题。_

**验证需求: 需求 5.2, 5.3**

### Property 5: 音乐播放状态在页面间保持

_对于任何页面，如果用户启动了背景音乐，当用户导航到其他页面时，音乐应该继续播放而不中断。_

**验证需求: 需求 6.2, 6.3**

### Property 6: 模块页面内容与原单页应用一致

_对于任何模块页面，该页面显示的内容应该与原单页应用中对应模块的内容完全相同。_

**验证需求: 需求 1.2**

### Property 7: 模块页面内的子导航链接正常工作

_对于任何模块页面，页面内的锚点链接（如 `#starstone`）应该能够正确导航到对应的内容部分。_

**验证需求: 需求 1.3**

### Property 8: 返回顶部按钮在所有页面上正常工作

_对于任何页面，当用户向下滚动超过300像素时，返回顶部按钮应该显示；当用户点击按钮时，页面应该平滑滚动到顶部；当用户滚动到顶部时，按钮应该隐藏。_

**验证需求: 需求 10.1, 10.2, 10.3**

### Property 9: 图片滚动功能仅在主页显示

_对于主页，图片滚动组件应该存在并正常工作；对于模块页面，图片滚动组件不应该存在。_

**验证需求: 需求 7.1, 7.2, 7.3**

### Property 10: 导航链接在所有页面上都能正确跳转

_对于任何页面，点击导航栏中的任何链接都应该导航到对应的页面，且新页面应该正确加载。_

**验证需求: 需求 2.2, 3.2, 3.3**

## 错误处理

### 资源加载失败

**场景**: CSS、JavaScript或图片文件无法加载

**处理方式**:

- 在浏览器控制台输出错误日志
- 提供备用样式（内联CSS）确保页面基本可用
- 使用 `onerror` 事件处理图片加载失败

### 路径错误

**场景**: 相对路径不正确导致资源无法加载

**处理方式**:

- 在开发阶段使用本地服务器测试
- 使用绝对路径作为备选方案
- 在部署前进行完整的路径验证

### 状态恢复失败

**场景**: `localStorage` 不可用或被禁用

**处理方式**:

- 使用 `try-catch` 包装 `localStorage` 操作
- 提供默认状态作为备选
- 在浏览器控制台输出警告信息

### 音乐播放失败

**场景**: 音乐文件无法加载或播放被浏览器阻止

**处理方式**:

- 捕获 `play()` 返回的 Promise 拒绝
- 提供用户交互作为播放触发器
- 在控制台输出详细的错误信息

1. 实现导航栏高亮功能

## 部署考虑

### 服务器配置

- 使用本地服务器（如 `http-server` 或 `python -m http.server`）进行开发
- 确保服务器支持相对路径解析
- 配置正确的MIME类型

### 路径配置

- 如果部署到子目录，需要调整所有相对路径
- 考虑使用 `<base>` 标签统一管理基础路径
- 或使用JavaScript动态计算基础路径

### 缓存策略

- 为CSS和JavaScript文件添加版本号或哈希值
- 配置适当的缓存过期时间
- 使用CDN加速资源加载

## 总结

本设计方案通过以下方式实现了从单页应用到多页应用的改造：

1. **清晰的文件结构**: 将模块分离为独立的HTML文件，便于维护和扩展
2. **智能的路径管理**: 使用相对路径确保资源在任何位置都能被正确加载
3. **跨页面状态管理**: 使用localStorage和全局对象保持用户偏好设置
4. **一致的用户体验**: 所有页面共享相同的导航栏、样式和交互功能
5. **完善的测试策略**: 通过单元测试、属性测试和集成测试确保功能正确性

这个方案既保持了原有的设计风格和交互功能，又提供了更好的代码组织和可维护性。
